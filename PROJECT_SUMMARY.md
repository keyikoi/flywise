# FlyWise 航班助手 - 项目技术总结

## 一、项目概述

FlyWise 是一个智能航班搜索助手，通过整合多个数据源（FlyAI/飞猪、SerpAPI/Google Flights、自有数据），为用户提供航班搜索和价格趋势分析服务。

---

## 二、核心架构设计

### 2.1 数据源策略

```
┌─────────────────────────────────────────────────────────┐
│                    用户查询航班                          │
└─────────────────────────────────────────────────────────┘
                          ↓
        ┌─────────────────┴─────────────────┐
        ↓                                    ↓
   国内航线                            国际航线
   (出发地和目的地都是中国城市)           (其他所有情况)
        ↓                                    ↓
   ┌────────────┐                      ┌────────────┐
   │  航班数据   │                      │  航班数据   │
   │   FlyAI    │                      │  SerpAPI   │
   │  实时查询   │                      │  实时查询   │
   └────────────┘                      └────────────┘
        ↓                                    ↓
   ┌────────────┐                      ┌────────────┐
   │  价格历史   │                      │  价格历史   │
   │  自有数据   │                      │  SerpAPI   │
   │  470k 条    │                      │  60 天     │
   └────────────┘                      └────────────┘
```

### 2.2 国内/国际航线判断逻辑

**判断规则**：
```javascript
// 只有出发地和目的地都是中国城市时，才算国内航线
if (出发地是中国城市 && 目的地是中国城市) {
    使用 FlyAI  // 国内航线
} else {
    使用 SerpAPI  // 国际航线（包括亚洲国际）
}
```

**中国城市列表**（部分）：
- 机场代码：PEK, PVG, SHA, CAN, SZX, CTU, XIY, KMG, HGH, NKG, NBS...
- 城市中文名：北京，上海，广州，深圳，成都，西安，昆明，杭州，南京...

### 2.3 为什么用工程判断而不是 LLM？

| 维度 | 工程判断 | LLM 判断 |
|------|---------|---------|
| **准确性** | 100% 确定 | 可能幻觉 |
| **速度** | <1ms | 500-2000ms |
| **成本** | 免费 | 每次调用都有成本 |
| **可维护性** | 规则清晰，易调试 | 黑盒，难排查 |
| **一致性** | 永远一致 | 可能波动 |

**选择工程判断的原因**：
1. **这是确定性规则**：城市列表是固定的，不需要语义理解
2. **性能考虑**：每次查询都要判断，LLM 会增加显著延迟
3. **成本控制**：避免不必要的 API 调用
4. **可解释性**：规则清晰，便于调试和扩展

**LLM 的正确使用场景**：
- 意图识别（用户想查航班还是问天气）
- 参数提取（从自然语言中提取城市、日期）
- 回复生成（将结构化数据转为自然语言）

## 三、数据流详解

### 3.1 完整请求流程

```
用户输入："明天杭州到北京的机票"
         ↓
┌────────────────────────────────────────┐
│ Step 1: LLM 意图识别                    │
│ - 判断意图：flight_search              │
│ - 提取参数：origin=杭州，destination=北京  │
│ - 提取日期：departDate=2026-03-26      │
└────────────────────────────────────────┘
         ↓
┌────────────────────────────────────────┐
│ Step 2: 工程判断航线类型                │
│ isDomesticRoute("HGH", "PEK") = true   │
│ → 选择 FlyAI 数据源                      │
└────────────────────────────────────────┘
         ↓
┌────────────────────────────────────────┐
│ Step 3: 调用航班搜索 API                 │
│ flyai search-flight --origin HGH ...   │
│ 返回 10 个航班，价格 ¥580-658            │
└────────────────────────────────────────┘
         ↓
┌────────────────────────────────────────┐
│ Step 4: 获取价格历史                    │
│ 从自有数据加载 HGH-PEK 航线 34 天记录      │
│ 每天取最低价航班                         │
└────────────────────────────────────────┘
         ↓
┌────────────────────────────────────────┐
│ Step 5: LLM 生成回复                     │
│ 输入：航班列表 + 价格历史               │
│ 输出："杭州→北京 3 月 26 日 航班精选..."    │
└────────────────────────────────────────┘
         ↓
┌────────────────────────────────────────┐
│ Step 6: 前端渲染                        │
│ - 航班卡片列表                          │
│ - 价格趋势图表                          │
│ - 购买建议                              │
└────────────────────────────────────────┘
```

### 3.2 LLM vs 工程路径分工

| 任务 | 实现方式 | 原因 |
|------|---------|------|
| **意图识别** | LLM | 需要理解自然语言语义 |
| **参数提取** | LLM | 日期格式多样（"明天"、"3 月 26 日"、"下周三"） |
| **航线判断** | 工程 | 确定性规则，无需语义理解 |
| **数据源选择** | 工程 | 基于航线类型的固定映射 |
| **数据查询** | 工程 | 调用 API 获取结构化数据 |
| **数据格式化** | 工程 | 固定格式转换 |
| **回复生成** | LLM | 需要自然语言组织能力 |
| **图表渲染** | 工程 | 前端可视化逻辑 |

## 四、价格趋势数据处理

### 4.1 国内航线（自有数据）

**数据来源**：
- Excel 文件：`26 年春运客座率 (2 月 8 日采集).xlsx`
- 记录数：470,177 条
- 覆盖航线：6,457 条
- 日期范围：2026-02-08 ~ 2026-03-13（34 天）

**价格计算**：
```python
# 从舱等数据计算平均价格
平均价格 = 总收入 (Income) ÷ 销售数 (BK)

# 示例：
# {"Income": "15578", "BK": "45"}
# 平均价格 = 15578 ÷ 45 = ¥346/张
```

**每天价格选取**：
```javascript
// 同一天有多个航班时，取最低价
if (!grouped[date] || item.avgPrice < grouped[date].avgPrice) {
    grouped[date] = item;  // 保留最低价航班
}
```

### 4.2 国际航线（SerpAPI）

**数据来源**：
- SerpAPI Google Flights
- 价格历史：60 天左右
- 格式：`[[timestamp, price], ...]`

## 六、技术选型总结

### 6.1 架构决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 航线判断 | 工程规则 | 确定性、高性能、零成本 |
| 意图识别 | LLM | 语义理解能力强 |
| 国内数据 | 自有数据 | 真实交易数据，更准确 |
| 国际数据 | SerpAPI | 覆盖广，无需维护爬虫 |
| 前端框架 | React | 组件化，生态丰富 |
| 后端框架 | Node.js + Express | 轻量，与前端语言统一 |

## 八、总结

FlyWise 的核心设计理念：
1. **LLM 做擅长的事**：语义理解、自然语言生成
2. **工程做确定的事**：规则判断、数据查询、格式转换
3. **数据源互补**：国内用自有数据（真实），国际用 SerpAPI（覆盖广）
4. **可扩展架构**：新增数据源只需修改 `flightService.js`

这种设计保证了：
- ✅ 准确性（确定性规则 + 真实数据）
- ✅ 性能（工程判断 <1ms）
- ✅ 成本（避免不必要的 LLM 调用）
- ✅ 可维护性（清晰的职责分工）
